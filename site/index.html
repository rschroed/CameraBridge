CameraBridge
How it works v1 Example

Local-only macOS camera service

Local camera access on macOS through one local API.

CameraBridge is a signed menu bar app and local service with a small, versioned localhost API. Apps, scripts, and other local software can use it instead of each carrying their own AVFoundation code.

  • Explicit camera permission state and recovery path
  • Localhost API for device and session control

Small surface area. Low drama.

Why this exists

Keep AVFoundation out of every host app.

01

Native camera code is repetitive.

Local apps and tools should not each rebuild camera discovery, permission handling, and session lifecycle in native macOS code.

02

Permission state should stay explicit.

CameraBridge centralizes camera permission visibility and request flow in one trusted local process instead of scattering it across individual clients.

03

Local software needs one clear API.

Apps, scripts, and local software can talk to one localhost API instead of each talking to AVFoundation directly.

How it works

A local service between your app and macOS.

Your app → CameraBridge → AVFoundation → macOS camera

Clients speak HTTP. CameraBridge handles camera access, and the menu bar app handles onboarding and status. macOS still controls permissions. That gives apps one local API instead of custom camera code in each client.

What is in v1

Small, explicit capabilities.

Runtime surface

  • GET /health
  • GET /v1/permissions
  • Device discovery
  • Session start, stop, and device selection
  • Still photo capture

Local product shape

  • Signed CameraBridgeApp bundle for install and lifecycle
  • Bundled camd daemon managed by the app
  • Menu bar onboarding, status, and permission recovery
  • Single active camera session model
  • Implicit ownership via session start in v1
  • Structured logs and local artifact metadata
  • Versioned localhost API under /v1

Trust model

Local-first constraints, stated clearly.

What CameraBridge is

  • macOS only
  • localhost only
  • Read-only endpoints may stay open in early v1
  • Bearer-token protected for mutating endpoints
  • No separate claim or release endpoints in v1
  • Single-user machine assumption

What CameraBridge is not

  • Not remote access
  • Not cloud sync
  • Not a browser permission bypass
  • Not a general media pipeline

Quick example

From device inventory to a still capture.

Read-only discovery

curl -s http://127.0.0.1:8731/v1/devices

Token-protected capture flow

TOKEN="$(cat ~/Library/Application\ Support/CameraBridge/auth-token)"

curl -s -X POST http://127.0.0.1:8731/v1/session/select-device \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"device_id":"YOUR_DEVICE_ID","owner_id":"client-1"}'

curl -s -X POST http://127.0.0.1:8731/v1/session/start \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"owner_id":"client-1"}'

curl -s -X POST http://127.0.0.1:8731/v1/capture/photo \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"owner_id":"client-1"}'

Readable by humans. Legible to agents.

The shipped path is small but practical: discover a device, use the local bearer token from the installed app, start the session, and capture a still image. If permission needs attention, local macOS clients can send people to the app through the documented `camerabridge://permission` flow. The repo also includes a minimal Python first-capture example.

Next step

If you're using CameraBridge from another app, install the signed app bundle first, use the menu bar app to start the service and handle permission recovery, then point your local client at the current v1 localhost API.
