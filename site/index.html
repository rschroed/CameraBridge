CameraBridge

Local-only macOS camera service

Local camera access on macOS, through one narrow localhost boundary.

CameraBridge exposes AVFoundation through a small, versioned local API so apps, scripts, and local software can use the camera through one deliberate macOS service instead of embedding native glue repeatedly.

Why this exists

Keep AVFoundation out of every host app.

01

Native camera glue is expensive.

Local apps and tools should not each rebuild camera discovery, permission handling, and session lifecycle in native macOS code.

02

Permission state should stay explicit.

CameraBridge centralizes camera permission visibility and request flow in one trusted local process instead of scattering it across individual clients.

03

Local software needs a stable boundary.

Apps, scripts, and local software can talk to one narrow localhost API rather than embedding AVFoundation directly.

How it works

A narrow service between your app and macOS.

Your app → CameraBridge → AVFoundation → macOS camera

Clients speak HTTP. CameraBridge owns native camera interaction. macOS still owns permissions. The result is a predictable local boundary instead of app-specific camera code.

What is in v1

Small, explicit capabilities.

Runtime surface

  • GET /health
  • GET /v1/permissions
  • Device discovery
  • Session start, stop, and device selection
  • Still photo capture

Local product shape

  • camd daemon and CLI entrypoint
  • Menu bar app for onboarding and status
  • Single active camera session model
  • Implicit ownership via session start in v1
  • Structured logs and local artifact metadata
  • Versioned localhost API under /v1

Trust model

Local-first constraints, stated clearly.

What CameraBridge is

  • macOS only
  • localhost only
  • Read-only endpoints may stay open in early v1
  • Bearer-token protected for mutating endpoints
  • No separate claim or release endpoints in v1
  • Single-user machine assumption

What CameraBridge is not

  • Not remote access
  • Not cloud sync
  • Not a browser permission bypass
  • Not a general media pipeline

Quick example

Health and permissions over local HTTP.

curl http://127.0.0.1:PORT/health

{
  "status": "ok"
}

curl http://127.0.0.1:PORT/v1/permissions

{
  "status": "not_determined"
}

Readable by humans. Legible to agents.

The shipped v1 surface stays intentionally small: health, permission visibility, device discovery, session control, and still capture. Mutating routes use bearer-token protection, with ownership established implicitly when a client starts the session.
