# CameraBridge v1 Roadmap

## 1. Overview

CameraBridge v1 focuses on a **minimal, reliable local camera service** for macOS.

The goal is to prove the core model:

> A local daemon (`camd`) can expose native macOS camera capabilities over a simple localhost API.

The shipped v1 slice is intentionally narrow. It prioritizes:
- clarity of architecture
- stability of the API contract
- a complete end-to-end loop

---

## 2. Definition of v1

A user can:

1. Install and run CameraBridge
2. Start the service from `CameraBridgeApp`
3. Verify the service is running (`/health`)
4. Inspect camera permission status
5. List available camera devices
6. Start a session
7. Capture a still image
8. Access the captured artifact locally

This flow is the current bar for the shipped v1 first-capture slice.

---

## 3. v1 Feature Scope

### Core API

- `GET /health`
- `GET /v1/permissions`
- `POST /v1/permissions/request`
- `GET /v1/devices`
- `GET /v1/session`
- `POST /v1/session/start`
- `POST /v1/session/stop`
- `POST /v1/session/select-device`
- `POST /v1/capture/photo`

---

### Core Capabilities

#### Permissions

- Read permission state
- Request camera permission via `CameraBridgeApp`
- In the shipped slice, `camd` reports live AVFoundation permission status used for session preconditions
- `POST /v1/permissions/request` stays API-first and returns guided next-step data when `CameraBridgeApp` must request access

#### Device Discovery

- Enumerate available camera devices
- Provide stable device identifiers and basic metadata

#### Session Lifecycle

- Single active session model
- Explicit start/stop semantics
- No multi-client arbitration beyond basic ownership
- No separate session claim/release endpoints in v1
- Successful session start establishes implicit ownership until stop or session end
- Mutating session and capture requests return a conflict when another client owns the active session

#### Still Capture

- Capture a single image
- Persist to local artifacts directory
- Return metadata (path, timestamp, device)

---

### Runtime

- `camd` daemon:
  - localhost HTTP server
  - readable startup and failure logging
  - runtime config + token management
  - runtime metadata for app-visible connection details

- Default binding:
  - `127.0.0.1`
  - fixed or configurable port

### Auth And Ownership

- Read-only endpoints remain unauthenticated in the shipped localhost-only slice
- Shipped mutating endpoints require a bearer token or equivalent local secret
- `POST /v1/permissions/request` is token-protected, does not create session ownership, and does not trigger the macOS prompt from `camd`
- when permission is still `not_determined`, the route returns guided next-step data telling callers to use `CameraBridgeApp`
- `POST /v1/session/start` acquires implicit session ownership on success
- `POST /v1/session/stop`, `POST /v1/session/select-device`, and `POST /v1/capture/photo` require the current session owner
- v1 uses explicit error categories for unauthorized, ownership-conflict, and invalid-state requests

---

### App Shell (Minimal)

- macOS menu bar app
- first-run onboarding
- permission status visibility
- app-supervised packaged daemon lifecycle with explicit start / stop / quit behavior
- developer-visible base URL, token path, log path, and captures path
- custom URL scheme is deferred until after v1

---

### Documentation

- README with quick start
- API reference for the shipped v1 endpoints
- quick-start guide
- minimal Python example for the first-capture flow
- release-readiness smoke test with a recorded real-hardware pass

---

## 4. Explicit Non-Goals (v1)

These are intentionally excluded from v1:

### Media / Capture

- video recording
- audio/microphone support
- advanced camera controls (exposure, focus, etc.)

### Streaming

- WebSocket preview transport
- remote streaming
- high-performance video pipelines

### System Complexity

- multi-session or multi-user support
- advanced client arbitration
- plugin system
- cross-platform support

### Network / Cloud

- remote access
- cloud sync
- authentication beyond local token model

### File Handling

- HTTP file serving layer
- asset management beyond basic local storage

---

## 5. Design Constraints

v1 must maintain:

- macOS-only implementation
- localhost-only access
- single active camera session
- strict separation of:
  - Core (camera logic)
  - API (HTTP interface)
  - Daemon (process + wiring)
  - App (UI only)

---

## 6. Milestones

### Milestone 1 - First Useful Loop (Shipped)

- daemon starts
- `/health` works
- permission status works
- devices listed

---

### Milestone 2 - Camera Control (Shipped)

- permission request
- session start/stop
- device selection

---

### Milestone 3 - Capture (Shipped)

- still image capture
- artifact storage + metadata

---

### Milestone 4 - Usability (Partially Shipped)

- menu bar app
- onboarding flow
- basic docs and Python example
- release-readiness smoke test documentation
- custom URL scheme remains deferred

---

## 7. Success Criteria

CameraBridge v1 is successful if:

- the full capture flow works end-to-end
- API behavior matches documentation
- no camera access requires embedding native code
- tests pass without requiring physical hardware
- the system is understandable by a new developer in under 10 minutes

---

## 8. What Comes After v1 (Not Committed)

Deferred areas after the shipped v1 first-capture slice (not guaranteed):

- video recording
- preview streaming improvements
- richer device controls
- improved auth model
- cross-platform exploration

These should only be considered after v1 is stable.

---

## 9. Guiding Principle

When in doubt, prefer:

> smaller, clearer, and more reliable

over:

> more features, more flexibility, or more abstraction

CameraBridge v1 should feel like:
- a **missing primitive**
- not a platform
- not a framework
